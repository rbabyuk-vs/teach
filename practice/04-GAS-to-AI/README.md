## Що таке Gemini CLI

> Gemini CLI — це круто, бо він працює прямо з вашим кодом у терміналі, а не в окремому чаті.
> Він бачить файли проєкту і розуміє контекст, тому підказки виходять точніші.
> Постійно копіювати код у браузер — це незручно і повільно, а Gemini CLI — інструмент, яким користуються реальні розробники.

## Що саме робити з Gemini CLI (щоб не було “копіпаст з чату”)

**Правило виробничого:** будь-які великі зміни робляться через CLI-агента.

Мінімальний чек-лист доказів:

* `PROMPTS.md` — 10–15 коротких промптів, які давали в Gemini CLI
* `CHANGELOG.md` — що змінили і навіщо
* історія комітів у git (мінімум 5–7 осмислених комітів)

Приклади задач для Gemini CLI:

* “Проаналізуй репозиторій GAS і склади план міграції до Vite PWA”
* “Створи структуру проєкту + базові файли”
* “Зроби компонент погодного віджета з fetch + рендер”
* “Додай service worker + offline cache”
* “Зроби Lighthouse checklist і виправ проблеми”

## Обрати рівень реалізації. Трек А для будь-кого, мінімум. Можна ообирати складнійший, буде окрема оцінка за це.

Календар — це найскладніше через авторизацію:

### Трек A (простий, 100% реально зробити)

* Календар: **iCal (.ics) публічне посилання** (read-only) або підготовлений `events.json`
* Погода: безключовий API (наприклад Open-Meteo)
* Деплой: GitHub Pages / Cloudflare Pages / Netlify

### Трек B (середній, “майже як прод”)

* Календар: **Google Calendar API read-only** з OAuth (можна через готовий шаблон)
* Погода: API + кешування
* Деплой: Firebase Hosting / Vercel / Netlify

### Трек C (advanced, для сильних)

* Фронт: PWA
* Бек: serverless (Cloudflare Workers / Firebase Functions / Cloud Run) як проксі + кеш + security
* Calendar інтеграція нормальна, з токенами
